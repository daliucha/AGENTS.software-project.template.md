## 1. 工程执行四原则

### 1.1 编码前思考

不要假设，不要隐藏困惑，必须呈现关键权衡。

- 明确说明假设：不确定时先指出，不要默默选择一种解释。
- 呈现多种解释：需求有歧义时，列出可能含义和影响。
- 适时提出异议：如果有更简单、更稳妥的方法，先说明理由。
- 困惑时停下来：关键业务规则、数据归属、破坏性操作不清楚时，先澄清。

### 1.2 简洁优先

用最少的必要代码解决当前问题，不为未提出的需求预留复杂性。

- 不添加要求之外的功能。
- 不为一次性逻辑创建抽象。
- 不添加未要求的灵活性、配置项、插件化。
- 不为明显不会发生的场景堆错误处理。
- 如果实现显得臃肿，先压缩到资深工程师能接受的复杂度。

检验标准：资深工程师会不会觉得这过于复杂？如果会，先简化。

### 1.3 精准修改

只碰必须碰的代码，只清理自己造成的混乱。

- 不顺手改相邻代码、注释、格式或命名。
- 不重构没坏、也不属于本次目标的模块。
- 匹配项目既有风格，即使个人偏好不同。
- 发现无关死代码时可以说明，但不要擅自删除。
- 删除因本次改动产生的无用导入、变量、函数、文件。

检验标准：每一行修改都能直接追溯到用户请求或验证失败。

### 1.4 目标驱动执行

把任务转化为可验证目标，循环验证直到达成。

- 修 bug：先理解或补充复现路径，再修到复现不再发生。
- 加验证：明确无效输入、期望错误、边界条件。
- 重构：确认行为不变，并运行相关测试或构建。
- 多步骤任务：先给短计划，每步带验证方式。

计划格式：

```md
1. {{步骤}} -> 验证：{{检查方式}}
2. {{步骤}} -> 验证：{{检查方式}}
3. {{步骤}} -> 验证：{{检查方式}}
```

### 3.1 链接引用规范

可以在本文件中引用外部链接和项目内文件链接，但必须遵守：

- 外部链接适合标注来源、官方文档、设计规范、API 文档入口；关键业务规则不要只放在外部链接里。

- 项目内文件优先使用相对路径，方便复制项目后继续有效。
- 涉及账号、密钥、数据库连接、客户隐私的文件不要作为公开模板示例。

- 项目内文档按任务需要读取；不要为了小改动读取全部文档。
- 外部链接只在需要确认官方行为、第三方 API、平台规则、库版本差异时读取。
- 如果用户明确要求“先看/核对/确认某文档或链接”，必须读取。
- 如果本地文档与已落地代码冲突，优先看已落地实现，并说明冲突。

## 4. 通用开发约束

- 不引入 mock 回退，除非项目明确要求；默认走真实服务或明确的本地离线模式。
- 不修改用户已有变更，除非用户明确要求。
- 不做无关重构、格式化、依赖升级、目录迁移。
- 不擅自改端口、API 基址、鉴权、安全头、数据迁移策略。
- 新增接口或字段时，同步检查客户端 API 封装、类型定义、页面/Store、后端 README 或接口文档。
- 涉及权限、状态机、数据归属、导出/下载、支付、删除、同步时，必须做对象级校验和回归验证。

## 5. 软件安全基线

本章是跨技术栈的默认安全底线。具体项目如果不需要某些平台，可删除对应小节；如果项目安全要求更高，应在本章继续补充。

### 5.1 通用安全原则

- 密钥、token、数据库连接、客户数据不能写进代码、日志、模板、截图或公开文档。
- 所有外部输入都不可信：前端表单、移动端请求、本地文件、导入数据、第三方回调、Webhook 都必须校验。
- 权限必须由后端或可信本地权限层强校验，UI 隐藏按钮不能视为安全。
- 账号、API token、数据库账号、云资源权限都按最小权限配置。
- 关键动作必须留痕：登录、失败、锁定、退出、重置密码、权限变更、删除、导出、审批、支付、同步。
- 错误信息不要泄露堆栈、SQL、内部路径、密钥、用户枚举线索。
- 新增依赖或升级依赖时，要考虑供应链风险、维护状态、许可证和体积影响。

### 5.2 Web 安全

- 不直接渲染不可信 HTML；确需富文本时必须做白名单清洗。
- Cookie 鉴权时必须考虑 CSRF；Bearer Token 鉴权时必须考虑 token 存储与 XSS 风险。
- CORS 不能无脑使用 `*` 搭配凭证；生产环境必须限制来源。
- CSP、iframe、文件预览、安全头要同时考虑安全与业务可用性。
- 下载、导出、附件预览必须鉴权、限流、审计，不开放永久裸链。
- 登录、验证码、刷新 token、敏感写接口应有限流或风控策略。

### 5.3 iOS / Android / 鸿蒙安全

- 敏感数据不能明文存普通本地存储；优先使用 Keychain、Keystore、系统安全存储或项目指定加密方案。
- 网络请求必须使用 HTTPS；不要为了调试长期关闭证书校验或 TLS 校验。
- 权限申请必须最小化，并向用户说明用途。
- 日志、崩溃上报、埋点不能输出 token、手机号、身份证、地址、业务敏感字段。
- 本地数据库如果包含敏感数据，要考虑加密、备份排除、迁移安全。
- App 被反编译是现实，核心鉴权、支付、越权控制必须放在服务端或可信环境。

### 5.4 单机与离线安全

- 本地文件导入要防畸形文件、路径穿越、超大文件和异常编码。
- 本地数据库要有迁移、备份、恢复和异常退出一致性策略。
- 离线授权、许可证、本地管理员能力必须有明确边界。
- 同步冲突不能导致越权覆盖、数据污染或旧数据覆盖新数据。
- 导出文件可能含敏感信息时，要提示用户并控制导出范围。

### 5.5 后端服务安全

- 鉴权、RBAC、对象级权限、租户隔离必须在后端强制执行。
- 状态机和关键业务动作不能只靠前端控制。
- ORM 查询也必须防越权查询；禁止先查全量再在业务层随意过滤敏感数据。
- 文件上传校验大小、MIME、扩展名、数量、对象归属，必要时增加内容扫描。
- 下载、预览、导出使用短效授权或签名 URL，并记录审计。
- 登录失败限制、Refresh Token 轮换、禁用用户立即失效应按项目风险落地。
- 数据库迁移、种子脚本、管理接口不能暴露生产风险。

### 5.6 AI 协作安全规则

- AI 不得把真实密钥、账号密码、数据库连接、客户隐私写进回答或文件。
- AI 不得为方便调试长期关闭鉴权、CORS、TLS、证书校验、安全头或对象级权限。
- AI 不得引入不必要依赖，尤其是低质量、无人维护或用途不清晰的包。
- 改登录、鉴权、权限、支付、导出、删除、同步、部署安全配置时，必须先说明风险和验证计划。
- 发现无关安全问题时可以指出，但不要顺手大改无关模块；除非用户要求修复。

### 5.7 安全触发阅读规则

- 改登录、鉴权、token、会话：必须先读 `{{认证与安全文档}}`
- 改权限、租户隔离、数据范围：必须先读 `{{权限边界文档}}`
- 改上传、下载、导出、文件预览：必须先读 `{{文件安全规则}}`
- 改部署、环境变量、CORS、CSP、安全头：必须先读 `{{部署/运维/安全头文档}}`
- 改支付、资金、隐私、删除、同步：必须先给风险说明和验证计划

## 6. 前端与客户端工程规范

本章适用于 Web、iOS、Android、鸿蒙、跨端、桌面端、单机离线应用。具体项目可删除无关平台小节，但应保留“架构、状态、网络、数据、体验、验证”这些横切规则。

### 6.1 架构与分层

- UI 层只负责展示、交互和局部状态，不直接散落业务规则、权限判断和接口细节。
- 业务状态、远程数据、本地缓存、表单草稿、全局会话要分清边界，不混在同一个对象里。
- 网络访问、错误映射、鉴权续期、环境切换应集中封装，不在页面/组件里重复拼接。
- 平台能力如相机、定位、文件、推送、蓝牙、剪贴板、系统分享应通过适配层调用，避免业务代码直接耦合系统 API。
- 公共组件只在真实复用出现后抽取；不要为了一个页面提前建设组件库。

### 6.2 状态管理与数据流

- 明确哪些状态来自服务端、哪些来自本地、哪些只是 UI 临时状态。
- 服务端状态要有加载、成功、空数据、错误、过期、刷新中的表现。
- 表单编辑应有初始值、脏状态、保存中、提交中、失败恢复和离开确认。
- 跨页面共享状态必须定义失效时机，避免用户切换账号、租户、环境后沿用旧数据。
- 对关键业务动作使用明确的提交状态，防止重复点击、重复提交和竞态覆盖。

### 6.3 网络与接口

- API/SDK 层统一处理 baseURL、鉴权头、token 刷新、超时、取消、重试、错误消息。
- 重试只用于幂等或明确可重试的请求；创建、支付、审批、删除、同步等动作不能盲目重试。
- 弱网、断网、超时、服务端 401/403/409/422/500 等状态必须有可理解的用户反馈。
- 客户端类型定义要和接口契约同步；新增字段、删除字段、枚举变化必须检查页面、Store、缓存和测试。
- 不在客户端信任服务端未授权的数据；也不把客户端校验当成最终安全边界。

### 6.4 UI、交互与可访问性

- 优先使用项目指定组件库或设计系统，不自造第二套交互和视觉语言。
- 页面布局、间距、按钮层级、表单校验、弹窗确认、空状态、错误状态保持一致。
- 危险操作必须二次确认，例如删除、撤回、驳回、办结、重置密码、清空数据。
- 按钮显隐只做体验优化，真正权限必须由后端或可信权限层强校验。
- 文案要说明用户当前能做什么、为什么失败、下一步怎么处理，避免暴露技术字段。
- 必须考虑键盘可达、焦点状态、屏幕阅读器语义、颜色对比、动态字体或系统字号。
- 响应式和适配要覆盖目标设备尺寸、安全区、横竖屏、折叠屏或窗口缩放要求。

### 6.5 表单、文件与本地数据

- 表单字段改动必须验证新建、编辑回显、再次提交、旧数据兼容和空值处理。
- 日期、金额、数字、时区、枚举、多选、动态列表、富文本要有统一格式化和解析规则。
- 文件上传、下载、预览、导入、导出必须有类型、大小、数量、权限、失败恢复校验。
- 本地缓存和本地数据库必须有版本、迁移、清理、备份或恢复策略。
- 离线草稿和同步队列要记录来源、时间、版本和冲突处理策略。

### 6.6 性能与可靠性

- 首屏、列表滚动、图片/文件预览、大表单、大数据量场景要避免明显卡顿。
- 长列表、媒体资源、地图、图表、大文件应按需加载、分页、虚拟化或压缩。
- 关键异步流程要处理取消、重入、重复点击、后台恢复、页面销毁后的回调。
- 错误边界或兜底页面要能阻止局部异常拖垮整个应用。
- 日志、埋点、崩溃上报要能帮助定位问题，但不能记录敏感信息。

### 6.7 平台专项

- Web：路由守卫必须处理未登录、无权限、首次改密、token 失效；文件预览要验证 CORS、CSP、iframe、安全头。
- iOS：处理 App 生命周期、安全区、动态字体、深色模式、后台任务、权限拒绝；敏感信息优先使用 Keychain。
- Android：处理 Activity/Fragment/Compose 生命周期、旋转、进程重建、运行时权限、后台限制；本地结构化数据变更要写 migration。
- 鸿蒙：处理 Ability 生命周期、ArkUI 状态更新、`module.json5` 权限、Preferences/relationalStore 数据变更和真机差异。
- 跨端：平台差异必须集中适配；不要把大量平台判断散落到业务页面。
- 单机/离线：明确无网可用范围、本地数据一致性、导入导出、同步冲突和异常退出恢复。

## 7. 后端服务规范

### 7.1 分层

- Controller 只做路由、DTO、当前用户提取。
- Service 承载业务规则、事务、状态机、对象级权限。
- Repository/ORM 层负责数据访问，不绕过业务校验。
- DTO 必须校验输入，禁止信任客户端字段。
- 公共工具只在出现真实复用时抽取。

### 7.2 鉴权与数据范围

- 接口必须强鉴权，前端隐藏按钮不能代替后端校验。
- 对象级权限必须用当前用户、角色、机构/租户/企业、对象归属、任务归属共同判断。
- 列表可见、详情可见、可编辑、可处理必须分离。
- 查询接口不能返回全量后让前端自行过滤敏感数据。
- 只读角色对写接口一律拒绝。

### 7.3 状态机与事务

- 状态变更必须校验当前状态、操作者、任务归属和版本。
- 状态变更、任务变更、日志写入必须在同一事务内完成。
- 重复提交、重复审批、重复支付等高风险动作需要幂等策略。
- 历史版本、审批意见、归档文件应只增不改。

### 7.4 文件、导出与归档

- 上传必须校验大小、MIME、扩展名、数量、对象归属。
- 下载、预览、导出必须校验权限，不能公开裸链。
- 短时效 token 或签名 URL 必须绑定文件和访问场景。
- 归档产物如 PDF、报表、凭证应记录版本、生成时间、hash、生成者。

### 7.5 审计与安全

- 登录、失败、锁定、退出、重置密码、权限变更、关键业务动作要留痕。
- 审计日志原则上只增不改。
- 登录、下载、导出、验证码、敏感写接口应有限流策略。
- 错误响应不要泄露栈、密钥、SQL、内部路径。

## 8. 功能规范模板

### 8.1 角色模型

用表格定义每个角色的权限和数据范围。

| 角色 | 能看什么 | 能改什么 | 能处理什么 | 数据范围 |
|---|---|---|---|---|
| `{{角色A}}` | `{{范围}}` | `{{范围}}` | `{{范围}}` | `{{租户/机构/本人/全局}}` |
| `{{角色B}}` | `{{范围}}` | `{{范围}}` | `{{范围}}` | `{{租户/机构/本人/全局}}` |

必须区分：

- 可见范围
- 可编辑范围
- 可处理范围
- 可删除范围
- 可导出/下载范围

### 8.2 核心状态机

| 状态 | 含义 | 允许操作 | 下一个状态 |
|---|---|---|---|
| `{{DRAFT}}` | `{{草稿/初始}}` | `{{保存/提交/删除}}` | `{{PENDING}}` |
| `{{PENDING}}` | `{{待处理}}` | `{{通过/退回/撤回}}` | `{{APPROVED/RETURNED}}` |
| `{{DONE}}` | `{{完成/归档}}` | `{{查看/下载}}` | `{{无}}` |

状态机要求：

- 不允许客户端直接指定最终状态。
- 不允许跳过前置节点，除非业务文档明确允许。
- 退回、撤回、重提、删除必须有明确条件。
- 历史处理意见、版本、归档结果必须保留。

### 8.3 列表与详情

- 列表接口必须按当前用户数据范围过滤。
- 详情接口必须复用同一套对象级权限。
- 列表出现某对象，不代表能编辑或处理该对象。
- 操作按钮由权限、状态、归属、任务共同决定。

### 8.4 表单与编辑回显

凡改动表单结构、接口字段、详情页、编辑页，必须验证：

- 新建能保存并提交。
- 编辑重新打开后字段完整回显。
- 日期、枚举、多选、动态列表、联系人/人员分组不会错位。
- 缺字段、旧数据、空数组、null 能被兼容处理。
- 再次提交不会覆盖无关字段或丢附件。

### 8.5 任务、审批、流转

适用于审批流、工单流、订单流、任务流。

- 当前待办必须绑定角色、机构、用户或任务池。
- 领取后应限制其他人处理，除非业务允许抢占或转派。
- 处理动作必须校验任务状态和操作者。
- 处理完成必须关闭当前任务并创建下一任务。
- 更正、退回、撤回必须留痕。
